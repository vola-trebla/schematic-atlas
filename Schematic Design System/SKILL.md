## The product

**Schematic Atlas** is a visual encyclopedia of MCP / AI tooling. Each
protocol gets exactly **one** page, generated by the canonical renderer at
`templates/protocol-page.jsx`. The catalog at `catalog/index.html` lists
every protocol. The catalog is the product; each page is an entry.

Do **not** build standalone product microsites with multiple pages, tabs,
nav, or drill-downs. That is explicitly the wrong direction. One protocol →
one page → one schematic explanation. The constraint is the point.

## To add a protocol from a real repo

The workflow is **hybrid**: enrich first, hand-author the soul second. There
is only one rendering path — the locked `templates/protocol-page.jsx`.

1. **Start from a flat input**: a name, a one-sentence purpose, a flat tools list, optional warnings + example. Same shape a typical MCP README naturally produces.
2. **Run the enricher** to get a 90%-complete rich config:
   - Interactive: open `templates/enrich-tool.html`, paste flat JSON, copy rich JSON.
   - CLI: `node templates/enrich-cli.mjs flat.json > rich.json`
   - From a README: `node templates/enrich-cli.mjs --readme README.md > rich.json`
3. **Copy** `templates/protocol-page.html` → `templates/examples/<name>.html` and paste the rich config into the `CONFIG` block.
4. **Hand-author the soul** — the 10% the enricher can't write:
   - Sharpen each `flow.steps[].question` (defaults are generic).
   - Add a real conditional `branch` with a custom red-stamp condition where the agent genuinely has a choice.
   - Rewrite each `groups[].blurb` to be tool-specific, not category-specific.
   - Rewrite the `example.captions` to point at the next call.
   - Override `highlight` if auto-detect picked wrong.
5. **Add the entry** to `ATLAS` in `catalog/atlas.jsx`. Optionally set `enriched: true` for internal analytics — it never renders.

**Hand-crafted illusion is the invariant.** The reader can't tell which
lines started from the script and which came from a human. Don't break
this by surfacing enrichment metadata in the UI.

## Design instincts (in priority order)

1. **The diagram is the page.** Most readers come for the flow. Spend your
   thinking budget here. The other sections are scaffolding.
2. **Annotations carry meaning, not decoration.** Each italic-blue question
   should answer _why an agent would call this_. Each red stamp should mark
   a real conditional branch the agent actually takes.
3. **Numbers stay numerals, code stays mono.** Tool names, args, packages,
   transports, file paths — all mono. Numerical stats — mono. Prose — hand.
4. **Highlight at most one word per page.** Usually the protocol's key noun.
5. **No emoji. No gradients. No drop shadows. No rounded corners** (except
   adhesive-label chips). No marketing voice. No "Get started in minutes".
